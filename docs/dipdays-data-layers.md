# DipDays Data Layers

DipDays persistence is intentionally separated into three explicit layers to keep large-scale ingestion stable and debuggable.

## 1) Source Snapshots (immutable inputs)

Examples:

- `data/raw/osm/`
- `data/raw/google/` (or provider-specific raw captures)
- `data/raw/reviews/`
- `data/raw/discovery/*.json`

Rules:

- Append-only snapshots of upstream provider payloads.
- Never mutate historical payloads in place.
- Preserve enough upstream metadata to replay canonicalization.

## 2) Canonical Records (normalized truth)

Examples:

- `data/processed/venues/*.canonical.json`
- Supabase `venues` table (synced canonical representation)

Rules:

- Single normalized representation per venue.
- Regenerated by canonicalization from source snapshots.
- Includes explicit provenance metadata:
  - `discovered_from`
  - `enriched_from`
  - `review_sources`
  - `last_canonicalized_at`

## 3) Derived Artifacts (recomputable outputs)

Examples:

- Evidence: `data/processed/evidence/`
- Scores: `data/processed/scores/`
- Pipeline receipts: `data/processed/pipeline/`
- Rankings / QA outputs: `data/processed/rankings*.json`, `data/review/`

Rules:

- Recomputable from canonical + source layers.
- Disposable when contracts change.
- Not source-of-truth for venue identity.

## Why this matters before scale

As venue counts grow into the thousands, this separation prevents drift:

- upstream providers can be re-fetched and diffed safely,
- canonical venues retain stable IDs + normalized shape,
- derived artifacts can be regenerated without corrupting source truth.
